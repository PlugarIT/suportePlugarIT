Status 11

O nome de arquivo especificado é inválido

The specified filename is invalid

O MicroKernel retorna este código de status nas seguintes situações:

  • O nome de arquivo especificado não segue as convenções ou o caminho é inválido.
    • -Verifique se o caminho do arquivo especificado não é muito longo. -Certifique-se de que o nome do arquivo e o caminho sejam válidos para o ambiente.
  • Quando estiver operando no ambiente cliente/servidor:
  • O aplicativo tentou abrir um arquivo que tem ".^^^" como extensão. Essa extensão é reservada para uso do MicroKernel durante operação contínua (somente sistemas de servidor podem usar operação contínua).
  • O buffer de dados para uma operação contínua Begin ou End não está corretamente configurado.
  • Você tentou carregar um arquivo remoto, mas as seguintes configurações do cliente estão incorretas:
    • -Client Access Use Local MicroKernel Engine está On -Client Access Use Remote MicroKernel Engine está Off. Para corrigir esta situação, abra o Pervasive Control Center na estação cliente, em Iniciar Programas Pervasive.SQL V8. No Control Center, clique duas vezes em Configuration, depois clique duas vezes em Client e Access. Ajuste Use Remote MicroKernel Engine para On. Para mais informações sobre os ajustes em Configuration, consulte o Advanced Operations Guide (Guia de Operações Avançadas).
  • Você tentou abrir um arquivo local usando um sistema de grupo de trabalho que não é o sistema de Gateway destinado ao arquivo. Vá para o diretório onde está o arquivo que você tentou abrir. Verifique se o arquivo ~pvsw~.loc desse diretório está marcado como somente leitura. Se estiver, remova o atributo de somente leitura
  • Quando estiver usando o arquivo de localização dinâmica com o sistema de grupo de trabalho:
  • Certifique-se de que o nome do arquivo de localização de segundo nível especificado no arquivo de localização de primeiro nível não tenha o mesmo nome que um diretório existente. Certifique-se também de que todos estejam na mesma unidade.
  • Certifique-se de que o arquivo de localização de segundo nível possa ser acessado pelo sistema.
  • Certifique-se de que todos os sistemas de grupo de trabalho que compartilham o arquivo de localização dinâmica tenham exatamente o mesmo mapeamento de unidade para o servidor onde estão os arquivos de dados.
  • Se você estiver acessando arquivos em um cliente DOS:
  • Não foi usado NET START FULL para a estação de trabalho Windows for Workgroups quando o DOS foi inicializado. Use NET START FULL para obter um redirecionador completo no modo de cliente do DOS.
    • BREQNT.EXE precisa de um redirecionamento completo. Na linha de comando do DOS, digite "net ver" e tecle . Esta é a lista dos componentes necessários para conectar uma estação de trabalho DOS a um servidor Windows NT: LSL.COM (driver de placa de rede) IPXODI.COM IFSHLP.SYS NET START FULL Todos podem ser carregados em memória alta usando emm386. NET START FULL é carregado em memória alta. NETX é necessário somente para conectar-se a um servidor NetWare.
  • Você tentou abrir um arquivo com nome longo em um volume NSS. O MicroKernel consulta os volumes usando chamadas do sistema operacional e depois carrega os drivers adequados aos espaços de nome associados encontrados. Neste caso, o MicroKernel estava sendo carregado antes que os volumes fossem montados, por isso não encontrou o suporte a nomes longos dos volumes NSS.
  • Chame os comandos MGRstart ou Bstart depois de carregar os volumes. Por exemplo:
    • LOAD NSS MOUNT ALL SYS:ETC\INITSYS.NCF MGRSTART ou BSTAR
  • Se você estiver usando o ambiente Microsoft Terminal Server:
  • Cerca de 5 usuários podem trabalhar em um aplicativo Windows em 2 servidores de terminal diferentes conectados a um servidor Windows NT 4.0 primário. A tentativa de ultrapassar este limite resulta nos códigos de status 11 e 35. Para solucionar este problema, consulte o artigo 190162 do Microsoft Knowledgebase, chamado "Terminal Server and the 2048 Open File Limitation".
